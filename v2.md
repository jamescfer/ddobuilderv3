# DDOBuilder V2 — Full Code-Layout Reference

> **Purpose.** A precise "where does everything live" map of the **V2** application
> (`DDOBuilder/`, a Windows MFC C++ DDO character planner, ~244 `.cpp` / ~274 `.h`)
> so that V3-parity work (`webapp/`, a React/TypeScript port) can be routed to the
> right code with confidence. Every subsystem is documented in detail in a section
> file under [`docs/v2ref/`](docs/v2ref/); this file is the **index + router + quick
> reference**. For the running V2↔V3 gap list and methodology, see
> [`PARITY_TODO.md`](PARITY_TODO.md).

## How to use this for V3 parity agents

1. **Find the subsystem** in the [Subsystem map](#subsystem-map) (8 sections, A–H).
2. **Find the exact file** in the [Master file index](#master-file-index) → it tells you
   the subsystem, the section doc to read, and the V3 counterpart file.
3. **Read the section doc** (`docs/v2ref/<X>_*.md`) for line-level detail: formulas,
   `File.cpp:line` refs, XML data sources, and the V3 counterpart for each class.
4. Cross-check the [Cross-cutting mechanics](#cross-cutting-mechanics-the-load-bearing-rules)
   (these rules touch many files) and the [Parity gaps](#parity-gaps-surfaced-during-this-mapping).

When fanning out agents, give each agent **one section** (or a slice of the
[Master file index](#master-file-index)) plus the relevant V3 target files.

## Cross-cutting mechanics (the load-bearing rules)

These rules are implemented once but affect dozens of files; get them right first.

1. **Effect bucketing & stacking** (`BreakdownItem.cpp:200-240,618-786`, section A/B).
   Every breakdown holds three buckets: `m_otherEffects` (computed), `m_effects`
   (feats/enhancements) and `m_itemEffects` (gear). `Total()` sums all three, but
   **only `m_itemEffects` (and anything flagged `ApplyAsItemEffect`) runs through
   `RemoveNonStacking`** — so "Highest-Only" bonus-type stacking applies to **gear
   only**; feat/enhancement bonuses always stack. → V3 models this with the
   `fromGear` flag in `bonus.ts`.
2. **Percentage effects** (`<Percent/>`, `DoPercentageEffects`). A flagged amount is
   a **percent of the stat's pre-percentage base total**, not a flat add (HP uniquely
   uses `DoAllPercentsAtOnce` for a single rounded sum). → V3 `percent` flag + the
   post-pass in `buildStatMap`.
3. **AType resolution** (`Effect.cpp:1030-1470`, section B). The `Amount`/`StackSource`/
   `Item`/`Cap` fields are read **differently per AType**. Ability-mod ATypes read the
   ability from **`StackSource`**, ignore `Amount`, and return the mod directly.
4. **Auto-acquisition** (`AutomaticAcquisition.cpp`, `Build.cpp:2493-2551`). Feats are
   granted when an `<AutomaticAcquisition>` entry's `.Met()` passes — the source of
   universal feats (Heroic Durability +30 HP, Completionist) that are in no class/race
   list. `IgnoreRequirements` lets them skip the feat's own train-requirements.
5. **Stance auto-activation** (`StancesPane.cpp`, `Build::IsStanceActive`). V2 auto-turns-on
   weapon-type and fighting-style stances (Two/Single Weapon Fighting, weapon type,
   Shield) from equipped gear; many enhancement effects gate on them. → V3 derives these
   in `buildStatMap` ctxStances.
6. **Inline `<Feat>` definitions.** Classes **and races** can define feats inline in their
   XML (e.g. Monk "AC Bonus: Wisdom", Drow "Drow Spell Resistance") and reference them by
   name. → V3 `loadFeats` folds **class**-inline feats but (gap) not race-inline.
7. **The `DL_*` macro pattern** (section H). Every V2 data class declares its fields once
   in an `X_PROPERTIES(_)` macro that expands into getters, members, SAX parse, validation
   and XML write — so an XML `<Element>` ⇄ C++ field mapping is found in the class header's
   `*_PROPERTIES` list.

## Parity gaps surfaced during this mapping

New findings from the section agents (beyond the items already in `PARITY_TODO.md`):

- **Race-inline `<Feat>`s not folded into V3's feat catalogue** — `loadFeats` unions
  class-inline feats but not race-inline ones (e.g. Drow Spell Resistance). (§D)
- **`Amount_SliderValueLookup`** indexes `Amount[]` by **ClassLevels(StackSource)** in V2
  (likely a V2 bug); V3 indexes by slider value. `SetBonusCount`/`FeatCount` are vector
  lookups in V2 but `base*count` approximations in V3. (§B)
- **Spell DC**: V2 uses the single `ClassCastingStat` and no universal term; V3 picks the
  highest-mod casting stat and adds universal `dc.All`/`dc.Spell`. V3's `METAMAGIC_KEYS`
  includes `EschewMaterials`, which is not a flag on V2's `Spell`. (§E)
- **Combat**: V2's attack-chain timeline simulator (`DPSPane` per-style evaluators) is an
  unfinished stub even in V2; V3's single-swing estimator is close to V2's *working* math.
  Remaining fidelity gaps: separate 19-20 crit multiplier, crit-only damage, true
  offhand-doublestrike-as-fraction-of-main, and fortification-vs-crit. (§F)
- **Several V2 breakdowns are stubs** (`BreakdownItemDice::SumDice` "NYI",
  `WeaponOtherDamageEffects` NYI) — target **live-game** behavior, not V2 output, for those. (§A)
- **Multi-life import (F1)** confirmed: `v2Import.ts:339-353` reads only the active
  life/build and drops the rest. (§C)
- **Synthesized feats**: V2 builds "Improved Heroic Durability (Class N)" and racial
  past-life / "Racial Completionist" feats **in code** (`Class.cpp:375-399`), not from XML. (§D)
